# وثيقة تحليل نظام تسجيل المستفيدين للبرامج التدريبية

## 1. ملخص المشروع

المنصة المطلوبة هي نظام تسجيل إلكتروني باللغة العربية وبتخطيط RTL لإدارة تسجيل الطلاب والطالبات في البرامج التدريبية داخل المديريات المعتمدة بمحافظة المحويت. يجب أن يدعم النظام التحكم بالمقاعد حسب الجنس والمديرية، ومنع التسجيل المكرر، وإغلاق البرامج الممتلئة، مع لوحة إدارة كاملة وتقارير وتصدير.

## 2. نطاق النظام

### ما يغطيه النظام

- عرض المديريات المعتمدة فقط
- عرض البرامج المرتبطة بالمديرية المختارة فقط
- التحقق من حالة البرنامج قبل عرض نموذج التسجيل
- التحقق من المقاعد العامة ومقاعد الذكور والإناث
- منع التسجيل المتكرر عبر رقم البطاقة أو رقم الهاتف
- تسجيل بيانات الطالب/الطالبة مع وقت الإرسال تلقائيًا
- لوحة إدارة لإدارة البرامج والمديريات والمقاعد والطلاب والتقارير
- سجل عمليات إدارية للتدقيق

### ما لا يغطيه في المرحلة الأولى

- الدفع الإلكتروني
- التكامل مع أنظمة خارجية حكومية
- تعدد اللغات خارج العربية
- تطبيق جوال مستقل

## 3. المستخدمون والصلاحيات

### الطالب أو الطالبة

- الدخول إلى صفحة التسجيل
- اختيار المديرية والبرنامج
- تعبئة البيانات المطلوبة
- إرسال الطلب فقط
- لا يملك صلاحية مشاهدة بيانات الآخرين أو تعديلها

### مدير النظام

- إدارة البرامج والمديريات والمقاعد
- عرض وتعديل وحذف بيانات الطلاب
- فتح وإغلاق التسجيل
- استخراج التقارير والتصدير
- مراجعة سجل العمليات

## 4. القواعد التشغيلية الأساسية

- التسجيل في برنامج واحد فقط لكل طالب.
- منع التكرار عند وجود رقم البطاقة أو رقم الهاتف مسبقًا.
- البرامج تظهر فقط بحسب المديرية المختارة.
- البرامج المغلقة لا تظهر للمستخدم.
- إذا وصلت المقاعد إلى الصفر، يتم إخفاء البرنامج أو إظهار رسالة اكتمال المقاعد.
- يتم تسجيل وقت وتاريخ الإرسال تلقائيًا.
- يتم احتساب المقاعد المتبقية حسب الجنس المختار.

## 5. تصور تجربة المستخدم

1. يختار المستخدم المديرية من قائمة معتمدة.
2. يعرض النظام البرامج المتاحة لتلك المديرية فقط.
3. عند اختيار البرنامج، يفحص النظام حالة البرنامج والمقاعد.
4. إذا كانت المقاعد غير متوفرة، تظهر رسالة اكتمال المقاعد ولا يظهر النموذج.
5. إذا كانت المقاعد متوفرة، يظهر نموذج البيانات الكامل.
6. عند الإرسال، يتم التحقق من عدم وجود تكرار ثم حفظ الطلب.

## 6. تصور البيانات

### الجداول الأساسية

- Students
- Programs
- Directorates
- Logs

### ملاحظات تحليلية

- يحتاج جدول الطلاب إلى قيد يمنع التكرار على رقم البطاقة ورقم الهاتف.
- يحتاج جدول البرامج إلى حقول للمقاعد الكلية ومقاعد الجنس الحاليّة.
- يفضل ربط الطلاب بالبرامج عبر Program_ID بدل النص الحر.
- سجل العمليات يجب أن يشمل المستخدم الإداري والوقت وبيانات IP.

## 7. متطلبات الأمان

- تشفير كلمات المرور.
- حماية لوحة الإدارة عبر مصادقة وجلسات آمنة.
- التحقق من المدخلات في الواجهة والخلفية.
- منع SQL Injection باستخدام الاستعلامات المهيكلة.
- تسجيل العمليات الإدارية للمراجعة.
- منع الوصول المباشر لقاعدة البيانات من الواجهة الأمامية.

## 8. متطلبات الواجهة

- واجهة عربية بالكامل.
- RTL.
- متجاوبة مع الهاتف والكمبيوتر.
- سريعة وواضحة وبسيطة.
- مناسبة لعمليات التسجيل السريعة وتقليل الأخطاء.

## 9. التحديات المتوقعة

- إدارة المقاعد المتزامنة عند كثرة الطلبات.
- منع التكرار بشكل موثوق حتى مع محاولات الإرسال المتكرر.
- إخفاء البرامج الممتلئة دون خلق تعارض بين العرض والتحقق في الخلفية.
- الحفاظ على أداء جيد عند ازدياد عدد الطلاب والبرامج.

## 10. التوصية التقنية

- الواجهة: React
- الخادم: Node.js مع Express.js
- قاعدة البيانات: MySQL
- التخزين: Cloud Storage للملفات المستقبلية
- الاستضافة: VPS

## 11. الخلاصة

النظام قابل للتنفيذ بشكل واضح إذا بدأنا بفصل 4 طبقات رئيسية: قاعدة البيانات، واجهات الإدارة، واجهة التسجيل العامة، ومنطق التحقق والخدمات الخلفية. الأولوية في التنفيذ يجب أن تكون لبناء نموذج بيانات قوي وقواعد تحقق صارمة قبل الانتقال إلى واجهات الاستخدام.
